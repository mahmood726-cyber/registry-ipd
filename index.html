Registry-Native Pseudo-IPD Reconstructor · ct.gov / AACT only · fully offline

Reconstructs pseudo individual-patient time-to-event data from ClinicalTrials.gov / AACT summary fields only — exact registry anchors, zero digitization error, full provenance to the NCT record.
Honest scope: AACT contains no Kaplan–Meier curves and no true IPD — only summary statistics. Output is always pseudo-IPD, never true IPD, and never produced for Tier C. The edge over digitization tools (IPDfromKM / KM-GPT / Guyot-on-image) is exact anchors on the rich-registry subset — not universal superiority.

Validation at a glance — how reliable is registry-native reconstruction? (AACT coverage · registry vs digitization · noise sensitivity)
Numbers below are embedded from this repo's validation runs — census_full_aact.py, headtohead.js, noise_sweep.js — against true patient-level data. Full interactive dashboard (gold standard, calibrated uncertainty, competing risks): validation-dashboard.html.

AACT coverage — what is reconstructable

Full 2026-06-01 snapshot: 76,067 trials with posted results, but a structured KM curve (≥3 timepoints) appears in only a few hundred — and zero post a structured number-at-risk row. Reconstructability is coverage-bound, not method-bound.

Registry-structured vs figure-digitization

Median |log HR| error on 25 true-IPD datasets. At equal density it's a near-tie (pixel noise costs little); a dense digitized figure beats few exact registry anchors — the registry path is anchor-density-limited, not noise-limited.

Digitization-noise sensitivity

Sweeping the digitization noise σ (survival, pp). Registry-exact is the flat noise-free reference; equal-density digitized crosses above it as noise grows, but a dense (K=25) digitized curve stays below registry across the whole range.
